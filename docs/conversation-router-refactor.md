# Conversation Service Router Refactor

This document records the target design for refactoring `stellaclaw` conversation handling into a packet-routed service model. It is an implementation design note, not a roadmap item.

## Goal

`ConversationRuntime` currently mixes routing, control-plane state, session lifecycle, channel rendering, cron execution, workspace coordination, host tool routing, and persistence in one large boundary.

The refactor goal is:

- Use one packet model for communication between conversation modules.
- Make `ConversationRouter` a routing layer, not a business coordinator.
- Make each service own its own protocol, state, and persistence boundary where practical.
- Keep conversation-control operations serialized in the owner thread.
- Avoid a second routing identity such as `LocalServiceActorId`.
- Avoid `ConversationEffect` as a second cross-module message model.

## Core Model

Inside a conversation, modules communicate by sending `ServicePacket`s to `ServiceAddress`es.

The router only understands:

- `source`
- `target`
- registered service handles

The router must not inspect business payload fields such as `action`, `request_id`, `kind`, or `echo_data`.

## Packet

The outer packet is deliberately minimal:

```rust
pub struct ServicePacket {
    pub source: ServiceAddress,
    pub target: ServiceAddress,
    pub payload: serde_json::Value,
}
```

There is no router-level `kind`.
There is no router-level `action`.
There is no router-level `request_id`.
There is no router-level `echo_data`.

Those fields may exist inside `payload` when a service protocol needs them. For example, `CronService`, `SessionService`, or `ManagedSessionService` can define their own payload schema containing `action`, `request_id`, `response_to`, or `echo_data`.

The packet model is:

```text
source address + target address + opaque payload
```

## Address

Every packet address includes the conversation id:

```rust
pub struct ServiceAddress {
    pub conversation_id: ConversationId,
    pub endpoint: ServiceEndpoint,
}
```

`conversation_id` is always present, including inside the owner thread. It is generated by the program, not by the LLM, and makes logs, validation, and future cross-conversation routing explicit.

`ServiceEndpoint` is the routing identity:

```rust
pub enum ServiceEndpoint {
    Service { name: ServiceName },
    Session { session_id: String },
    Channel { channel_id: String },
    Workspace,
}
```

There should not be a second local routing enum. In particular, do not introduce or retain a separate `LocalServiceActorId` for routing.

Examples:

```text
web-main-000001 : service(conversation_control)
web-main-000001 : service(session)
web-main-000001 : service(cron)
web-main-000001 : service(managed_session)
web-main-000001 : service(memory)
web-main-000001 : service(skill)
web-main-000001 : service(tool_binary)
web-main-000001 : session(web-main-000001.foreground)
web-main-000001 : session(web-main-000001.background.0001)
web-main-000001 : channel(web)
web-main-000001 : workspace
```

`ServiceName` is validated:

- ASCII letters only.
- `_` and `-` are allowed.
- Empty names are invalid.
- Digits, dots, path separators, spaces, and other punctuation are invalid.

`Session { session_id }` and `Channel { channel_id }` are not service names. They can use their own id rules.

## Service

A service is an addressable participant inside a conversation.

At minimum, every service has:

```rust
pub trait ConversationService {
    fn endpoint(&self) -> ServiceEndpoint;
}
```

There are two delivery styles.

Direct services run on the conversation owner thread:

```rust
pub trait DirectService: ConversationService {
    fn handle_packet(
        &mut self,
        runtime: &mut ConversationRuntime,
        packet: ServicePacket,
    ) -> anyhow::Result<()>;
}
```

Mailbox services own an inbox and may run in an async task, worker thread, or wrapper around an existing runtime:

```rust
pub trait ActorService: ConversationService + Send + 'static {
    fn run(
        self,
        ctx: ServiceContext,
        inbox: Receiver<ServicePacket>,
    ) -> anyhow::Result<()>;
}
```

Service context contains the minimum routing capability:

```rust
pub struct ServiceContext {
    pub self_address: ServiceAddress,
    pub router_tx: Sender<ServicePacket>,
}
```

A service should:

- Parse only packets addressed to itself.
- Treat packet payload as its own protocol.
- Own its private state where practical.
- Emit packets to other services instead of mutating their state directly.
- Persist its own durable data where practical.

## Service Protocol Ownership

Protocol ownership depends on whether the message is one-way or two-way.

One-way messages are owned by the sender. A sender that emits a fire-and-forget event or notification defines that message shape.

Examples:

- `SessionService` owns session output messages.
- `SessionService` owns session lifecycle messages.
- `CronService` owns cron notification messages.
- `ConversationControlService` owns conversation status/control messages.
- `ChannelService` owns channel input messages emitted from external transports.

Two-way request/response protocols are owned by the request receiver. The service that receives the request defines both the request payload and the response payload.

Examples:

- `CronService` owns `cron_task_create` request and `cron_task_create` response.
- `SessionService` owns `start_session` request and `start_session` response.
- `WorkspaceService` owns `read_file` request and `read_file` response.
- `TerminalService` owns `create_terminal` request and `create_terminal` response.

There is no shared payload envelope layer. If two services intentionally use the same action name or the same Rust struct, that is a local code reuse choice, not a router or architecture concept.

The router still sees only:

```text
source + target + payload
```

It does not know whether a packet is one-way, request, response, progress, or error.

## Service Handle

The router registration table maps an endpoint to a service handle:

```rust
pub enum ServiceHandle {
    Direct,
    Mailbox(Sender<ServicePacket>),
}
```

`Direct` means the packet is returned to the owner thread for direct service invocation.

`Mailbox` means the router sends the packet to a service inbox.

The router does not store an actor id, host service id, or business service enum. The endpoint is the identity.

## Router

The router owns the route table:

```rust
pub struct ConversationRouter {
    routes: BTreeMap<ServiceEndpoint, ServiceHandle>,
}
```

Routing logic:

```rust
fn route(packet: ServicePacket) -> Result<RouterDispatch> {
    validate_same_conversation(packet.source, packet.target)?;
    validate_source_registered(packet.source.endpoint)?;

    match routes.get(&packet.target.endpoint) {
        Some(ServiceHandle::Direct) => Ok(RouterDispatch::Direct(packet)),
        Some(ServiceHandle::Mailbox(tx)) => {
            tx.send(packet)?;
            Ok(RouterDispatch::Delivered)
        }
        None => Err(UnknownTarget),
    }
}
```

Router responsibilities:

- Validate conversation scope.
- Validate source and target registration.
- Resolve target endpoint.
- Deliver the packet using the registered handle.

Router non-responsibilities:

- It does not inspect `payload.action`.
- It does not parse request/response shapes.
- It does not own cron/session/channel/workspace business state.
- It does not decide where session events should go.

## Registration

Conversation startup registers the stable service routes.

The target shape is to register service families, not every dynamic runtime instance:

```rust
service(conversation_control) -> Direct
service(session)              -> Direct or Mailbox
service(cron)                 -> Direct or Mailbox
service(managed_session)      -> Direct or Mailbox
service(memory)               -> Direct or Mailbox
service(skill)                -> Direct or Mailbox
service(tool_binary)          -> Direct or Mailbox
channel(web)                  -> Direct or Mailbox
workspace                     -> Direct or Mailbox
```

Dynamic details stay inside the owning service:

- `SessionService` knows whether `session(web-main-000001.background.0001)` exists and how to reach it.
- `ChannelService` knows how many subscribers or transport instances are behind `channel(web)`.
- `CronService` knows which cron tasks and cron-created background agents it owns.
- `WorkspaceService` decides whether an operation touches local state, fixed SSH state, or overlay state.

This keeps the router table stable after conversation startup in the common case.

If a future feature genuinely needs route-table mutation, only `ConversationControlService` should have router-admin capability, and it should run as a direct service on the owner thread.

## Conversation Control Service

`ConversationControlService` is a normal addressable service:

```text
service(conversation_control)
```

It is special only in capability, not in address shape.

It must be registered as `Direct`:

```rust
router.register(
    ServiceEndpoint::service("conversation_control")?,
    ServiceHandle::Direct,
);
```

It runs on the conversation owner thread because it can modify owner-owned control state and, when needed, the router table.

Primary responsibilities:

- Handle conversation-level control commands.
- Handle lifecycle/control events for sessions it creates or owns.
- Create and destroy sessions when requested by service protocols.
- Record and validate session reply bindings for sessions it creates or owns.
- Register or unregister routes if a future design needs dynamic routes.
- Persist conversation-level control state.

It should not become the default landing point for all business messages. Business packets should target the owning service directly.

`ConversationControlService` is the control-plane owner for sessions it creates or owns. It is not a global session lifecycle registry and not a session business-message router.

It owns questions such as:

- Does this session exist?
- Is this session running, stopped, failed, or destroyed?
- Which session is foreground?
- Does a session creation/destruction request need to update owner state or route state?

It should not decide:

- Where a cron-created background agent's final output should go.
- How a cron task result should be shown or stored.
- How a managed subagent result should be joined.
- Whether a background result should be forwarded to a foreground session or a channel.

Those decisions belong to the service that created the session and set its `reply_to`.

Examples:

- Foreground session events can target `service(conversation_control)` when conversation control created the foreground session.
- Cron-created background session events target `service(cron)`.
- A user-visible output packet targets `channel(web)`.
- A model/tool request for a session targets `service(session)` or a concrete `session(...)` address depending on the session protocol.

## Session Service

`SessionService` owns agent-server session routing.

It has three responsibilities:

- Session runtime adapter.
- Session event emitter.
- Session tool bridge adapter.

It should understand session ids and session bindings internally. The router should not need to register every background or subagent session as a separate route.

### Session Runtime Adapter

`SessionService` owns the mapping from `session_id` to the concrete agent-server client/runtime handle.

It handles:

- Starting a session.
- Stopping a session.
- Sending `SessionRequest` values to agent-server.
- Receiving `SessionEvent` values from agent-server.
- Maintaining `SessionBinding` metadata.

It does not decide how cron results, managed-session results, or user-visible messages should be interpreted.

Session runtime events are classified into two packet flows:

- `SessionRuntimeEvent`
- `ToolBridgeRequest`

Do not introduce a fourth generic "raw business event" category.

These names describe flow categories. They do not create a router-level packet kind. One-way session runtime event payloads are owned by `SessionService`; request/response flow payloads are owned by the request receiver.

`SessionService` inbound protocol includes session-control actions:

```text
source=service(cron)
target=service(session)
payload={ "action": "send_to_session", "payload": { "session_id": "...", ... } }
```

or, if the protocol keeps concrete session endpoints:

```text
source=service(cron)
target=session(web-main-000001.background.0001)
payload={ ... }
```

The recommended starting point is to route session commands to `service(session)` and carry `session_id` in payload. A later version may let `Session { session_id }` target the same `SessionService` family route, but it should not require per-session router registration.

Session events should not all go to conversation control. Each session has one reply target:

```rust
pub struct SessionBinding {
    pub session_id: String,
    pub reply_to: ServiceAddress,
}
```

Examples:

- Foreground session events can target `service(conversation_control)`.
- Cron-created background session events target `service(cron)`.
- Tool-created background/subagent events target `service(managed_session)`.

The service that creates a runtime sets `reply_to` and receives all events from that runtime. That service then decides what to persist, what to ignore, and what to forward to `channel(...)`, `service(session)`, or another service.

### Session Flow Categories

`SessionRuntimeEvent` packets describe anything emitted by the session runtime, including lifecycle, assistant output, tool result messages, final output, and runtime failure. They are one-way messages owned by `SessionService` and always target `binding.reply_to`:

```text
source=session(web-main-000001.background.0001)
target=service(cron)
payload={
  "action": "session_runtime_event",
  "payload": {
    "session_id": "web-main-000001.background.0001",
    "session_type": "background",
    "agent_id": "background_0001",
    "event": "started|ready|output|failed|stopped|destroyed",
    "messages": [ "optional ChatMessage serialized" ],
    "status": "optional",
    "final": false,
    "reason": "optional failure/stop reason"
  }
}
```

For a foreground session whose `reply_to` is `service(conversation_control)`, conversation control consumes the same `SessionService`-owned runtime event:

```text
source=session(web-main-000001.foreground)
target=service(conversation_control)
payload={
  "action": "session_runtime_event",
  "payload": {
    "session_id": "web-main-000001.foreground",
    "session_type": "foreground",
    "agent_id": null,
    "event": "output",
    "messages": [ "ChatMessage serialized" ],
    "status": "optional",
    "final": false
  }
}
```

The reply service may then forward user-visible output to `channel(web)` using a channel event message:

```text
source=service(conversation_control)
target=channel(web)
payload={
  "action": "delivery",
  "payload": {
    "delivery": "OutgoingDelivery-equivalent payload"
  }
}
```

`session_runtime_event` carries provider-neutral `ChatMessage` values when message content is present. It must follow the repository `ChatMessage` and `FileItem` conventions:

- Do not store provider-specific multipart or Responses item shapes in packet payloads.
- Use standard `FileItem` payloads for files.
- Tool results should prefer structured payloads.
- Channel-facing text should be rendered by the receiving display/service layer.

`ToolBridgeRequest` packets represent model tool/service calls. They target the service resolved by the tool/action registry. Because tool calls are request/response protocols, the request target owns both request and response payloads:

```text
source=session(web-main-000001.foreground)
target=service(cron)
payload={
  "action": "cron_task_create",
  "request_id": "...",
  "payload": { ... },
  "echo_data": {
    "session_id": "web-main-000001.foreground",
    "tool_call_id": "..."
  }
}
```

### Session Tool Bridge Adapter

When a model inside a session calls a host tool or service-backed action, `SessionService` acts as the bridge from agent-server to the service bus.

For example, a model call to `cron_task_create` becomes a packet:

```text
source=session(web-main-000001.foreground)
target=service(cron)
payload={
  "action": "cron_task_create",
  "request_id": "...",
  "payload": { ... },
  "echo_data": {
    "session_id": "web-main-000001.foreground",
    "tool_call_id": "..."
  }
}
```

The target service owns the request action protocol and response payload shape. `SessionService` should not parse cron, memory, skill, or tool-binary request payloads.

Tool/service action routing should come from a registry, not from ad hoc service logic inside `SessionService`.

Example mappings:

```text
cron_task_create   -> service(cron)
cron_task_update   -> service(cron)
memory_search      -> service(memory)
skill_create       -> service(skill)
tool_binary_ensure -> service(tool_binary)
start_background   -> service(managed_session)
```

The response returns as a packet to `service(session)`, but its business result shape is still defined by the original request receiver. `SessionService` only uses `request_id` / `echo_data` to match the response back to the agent-server tool call:

```text
source=service(cron)
target=service(session)
payload={
  "action": "tool_response",
  "request_id": "...",
  "payload": {
    "result": ...
  },
  "echo_data": {
    "session_id": "web-main-000001.foreground",
    "tool_call_id": "..."
  }
}
```

`SessionService` then converts the response back into the agent-server `ToolResultItem`.

## Cron Service

`CronService` owns cron scheduling and cron-created background work.

`CronService` should be registered as `Mailbox`. It owns its own scheduler loop and decides when tasks are due. It should not receive due tasks as typed owner commands, and the owner/manager should not send `cron_tick` packets for normal scheduling.

Cron task persistence belongs to `CronService`. The target persistent location is service-owned storage such as:

```text
services/<conversation_id>/cron/
```

Both script tasks and background-agent tasks are started by `CronService`:

- Script tasks execute inside `CronService`'s own task runner.
- Background-agent tasks are created by sending a request to `service(session)`.

Cron-created background sessions must be initialized with:

```text
reply_to = service(cron)
```

All runtime events from those sessions return to `service(cron)`.

`CronService` can then decide whether to:

- Update cron-owned task state or run state.
- Forward output to a foreground session.
- Forward user-visible output to a channel.
- Ignore or only log session output.

Cron does not need to persist full background-agent output by default. A background-agent cron task should have a task-level result destination policy that decides where useful results go. Script mode can define its own output behavior.

`CronService` owns the request/response protocols for:

```text
cron_tasks_list
cron_task_get
cron_task_create
cron_task_update
cron_task_remove
```

## Channel Service

Channels are addressable endpoints:

```rust
ServiceEndpoint::Channel { channel_id: String }
```

From the conversation point of view, a channel is a service endpoint. Its internals decide how `channel_id` maps to Web, Telegram, or other transport instances.

`Channel { channel_id }` is enough for this migration. Multi-subscriber WebSocket details should remain inside `ChannelService` until a concrete routing requirement appears.

`ChannelService` should be registered as `Mailbox`. It is an external I/O boundary and may perform transport sends, subscriber fanout, rendering, attachment preparation, or other work that should not block the conversation owner thread.

`ChannelService` has two responsibilities:

- Deliver server-originated UI/channel events to external clients.
- Deliver frontend HTTP/WebSocket responses back to the request or connection that initiated them.

It should not own workspace file operations or terminal runtime state. Those belong to `WorkspaceService` and `TerminalService`.

### Server-Originated Channel Events

`ChannelService` must preserve the current `ChannelEvent` capability set. In packet form, it should be able to receive and render the equivalents of:

```text
delivery
message_appended
processing
progress_feedback
conversation_updated
status
error
```

These are one-way messages. Their concrete message schemas are owned by the service that emits them, but `ChannelService` must support rendering the full capability range that the current channel layer exposes.

Current capability mapping:

- `delivery`: send a user-visible message, with optional `ChatMessage`, text, attachments, and options.
- `message_appended`: notify clients that a `ChatMessage` was appended to conversation history.
- `processing`: update typing/idle state.
- `progress_feedback`: update turn progress, plan state, activity, hint, done/failed state.
- `conversation_updated`: refresh conversation metadata/list state.
- `status`: render model, reasoning, sandbox, remote, workspace, background/subagent, and usage status.
- `error`: render structured channel-visible errors with scope, severity, code, detail, continuation state, and suggested action.

For example, foreground session output can be emitted by `SessionService` and consumed by `ChannelService`:

```text
source=session(web-main-000001.foreground)
target=channel(web)
payload={
  "action": "session_output",
  "payload": {
    "session_id": "web-main-000001.foreground",
    "session_type": "foreground",
    "agent_id": null,
    "messages": [ "ChatMessage serialized" ],
    "status": "optional",
    "final": false
  }
}
```

Cron or conversation-control messages can similarly target the channel as one-way display messages:

```text
source=service(cron)
target=channel(web)
payload={
  "action": "channel_message",
  "payload": {
    "text": "Cron task completed.",
    "attachments": [],
    "level": "info"
  }
}
```

Status/progress/error events should map to the existing channel event capabilities rather than collapse into one generic state message:

```text
source=service(conversation_control)
target=channel(web)
payload={
  "action": "status",
  "payload": {
    "status": "OutgoingStatus-equivalent payload"
  }
}
```

### Frontend Request Response Delivery

Frontend HTTP/WebSocket requests should also return through `ChannelService`.

The flow is:

```text
Frontend request
-> ChannelService transport adapter
-> ServicePacket to target service
-> target service handles the request
-> response packet to channel(web)
-> ChannelService writes response to the original frontend request/connection
```

For this, `ChannelService` maintains transport response state such as:

```text
request_id -> pending HTTP response sink
connection_id -> active WebSocket sink
```

`ChannelService` should accept transport response delivery actions such as:

```text
http_response
websocket_frame
websocket_close
```

Example workspace file request:

```text
source=channel(web)
target=service(workspace)
payload={
  "action": "read_file",
  "request_id": "web-http-123",
  "payload": {
    "conversation_id": "web-main-000001",
    "path": "src/main.rs",
    "offset": 0
  }
}
```

Workspace response returns through the channel:

```text
source=service(workspace)
target=channel(web)
payload={
  "action": "http_response",
  "request_id": "web-http-123",
  "payload": {
    "status": 200,
    "content_type": "application/json",
    "body": {
      "workspace_read_file_response": "WorkspaceService-defined response body"
    }
  }
}
```

The business response body is defined by `WorkspaceService`, because `read_file` is a `WorkspaceService` request/response protocol. The transport wrapper that writes it back to HTTP is defined by `ChannelService`.

Example terminal stream:

```text
source=channel(web)
target=service(terminal)
payload={
  "action": "attach_terminal",
  "request_id": "websocket-terminal-55",
  "payload": {
    "terminal_id": "terminal_0001",
    "offset": 0
  }
}
```

Terminal output returns through the channel connection:

```text
source=service(terminal)
target=channel(web)
payload={
  "action": "websocket_frame",
  "request_id": "websocket-terminal-55",
  "payload": {
    "encoding": "utf8",
    "data": "..."
  }
}
```

The terminal attach/request and terminal output business body are defined by `TerminalService`; the WebSocket frame delivery wrapper is defined by `ChannelService`.

### Frontend Ingress

External user input enters through the channel transport and is emitted as a one-way message owned by `ChannelService`, usually to `service(conversation_control)`:

```text
source=channel(web)
target=service(conversation_control)
payload={
  "action": "channel_input",
  "payload": {
    "message": "IncomingConversationMessage serialized"
  }
}
```

Other frontend APIs should target their owning services:

```text
workspace/file API -> service(workspace)
terminal API/stream -> service(terminal)
conversation/session controls -> service(conversation_control)
```

ChannelService is the transport boundary for those requests and responses, not the business owner.

## Workspace Service

`ServiceEndpoint::Workspace` is an addressable workspace capability endpoint, not a path.

The endpoint represents workspace-facing operations such as:

- Reading or mutating conversation workspace files.
- Operating on a fixed SSH remote workspace.
- Materializing local overlay files.
- Uploading, downloading, archiving, or previewing artifacts.
- Reporting progress to a channel endpoint.
- Returning structured results to a session endpoint.

The address should not expose local/remote implementation details. Fixed SSH mode should behave externally like a single overlay-filesystem-like workspace service.

## Service Protocols

The router should not standardize request ids, response shape, action names, or echo data globally.

Protocol documents should separate:

- One-way messages owned by the sending service.
- Request/response protocols owned by the request-receiving service.

Responses are still transported as `ServicePacket`s, but the response payload shape belongs to the service that defined the corresponding request.

Suggested protocol documents:

```text
stellaclaw/src/conversation/services/session/protocol.md
stellaclaw/src/conversation/services/cron/protocol.md
stellaclaw/src/conversation/services/channel/protocol.md
stellaclaw/src/conversation/services/workspace/protocol.md
stellaclaw/src/conversation/services/managed_session/protocol.md
stellaclaw/src/conversation/services/terminal/protocol.md
```

The service protocol can contain `action`, `request_id`, `reply_to`, `echo_data`, and response payload conventions if that service needs them. Those fields stay inside `payload`; the router does not inspect them.

## Conversation Effect

`ConversationEffect` should not be part of the target architecture.

It was useful in a function-call-oriented design to represent "the handler returned work for the owner to apply." In the packet model, it becomes a second cross-module message system.

Target rule:

```text
Cross-module communication uses ServicePacket only.
```

Control operations should be packets to `service(conversation_control)`.
Business work should be packets to the owning service.
User-visible output should be packets to `channel(...)`.

During migration, existing `ConversationEffect` code can be removed or temporarily treated as a private implementation helper, but it should not remain as a service abstraction.

## Migration Plan

1. Finalize `ServicePacket`, `ServiceAddress`, `ServiceEndpoint`, `ServiceHandle`, and `ConversationRouter`.
2. Replace `ConversationControl` endpoint with `service(conversation_control)`.
3. Remove `LocalServiceActorId` and any second routing identity.
4. Register stable service routes at conversation startup.
5. Implement direct dispatch for `service(conversation_control)`.
6. Route session-family endpoints to `SessionService` instead of registering every session instance as an independent route.
7. Move cron due execution behind `CronService`.
8. Route cron-created background agent events to `service(cron)`.
9. Route tool-created background/subagent events to `service(managed_session)`.
10. Convert user-visible output to packets targeting `channel(...)`.
11. Delete `ConversationEffect` from the service abstraction and replace remaining cross-module effects with packets.
12. Move service-owned state and persistence behind the owning service where practical.

The first implementation should preserve current user-visible behavior and should not change `config` or `workdir` schema unless a later step explicitly requires persisted session `reply_to` metadata.

## Current Implementation Note

The repository currently contains transitional code from an earlier iteration of this refactor. Some names and files may still reflect the previous compatibility approach, including local direct-dispatch helpers and `ConversationEffect`.

Those pieces should be treated as migration scaffolding, not the target design. The target design in this document is authoritative for the next refactor step:

- Endpoint is the service identity.
- Router stores endpoint-to-handle registrations.
- `conversation_control` is a direct service with router-admin capability.
- Business services communicate only by `ServicePacket`.
- `ConversationEffect` should be removed from the service abstraction.
