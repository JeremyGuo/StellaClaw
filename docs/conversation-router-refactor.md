# Conversation Service Router Refactor Draft

This note records the current refactor direction for `stellaclaw` conversation handling. It is a discussion draft, not a finalized roadmap item.

## Current Problem

`ConversationRuntime` currently owns the serial conversation boundary, but it also directly contains many concrete responsibilities:

- Channel input handling.
- Session event polling and dispatch.
- Host service bridge routing.
- Foreground/background/subagent session lifecycle.
- Channel event rendering and progress feedback.
- Durable `ConversationState` persistence.
- Workspace, remote, sandbox, cron, memory, skill, and tool-binary coordination.

This makes the conversation layer more than a router. It becomes a large coordinator with service-specific logic embedded in the same file.

## Target Model

Treat every participant as a service endpoint:

- Conversation control itself.
- Foreground/background/subagent sessions.
- Memory.
- Skill.
- Cron.
- Tool binary manager.
- Channel endpoints.
- Workspace-facing endpoints.
- Future dynamically registered services.

`Conversation` should become a single-threaded owner that contains:

- A router.
- A service registry.
- A conversation-control service.
- Durable `ConversationState`.
- Session binding/runtime handles.
- Workspace, remote, sandbox, and progress state.

The router and conversation-control service are conceptually separate, but should run in the same owner thread. This keeps route table updates, state mutation, and lifecycle changes serialized without locks.

## Minimal Packet

The router-level packet should stay deliberately small:

```rust
pub struct ServicePacket {
    pub source: ServiceAddress,
    pub target: ServiceAddress,
    pub payload: serde_json::Value,
}
```

The router should not require outer `kind`, `request_id`, `action`, or `echo_data` fields.

Those fields can exist inside `payload` when a service protocol needs them. For example, a session tool-call protocol may put `request_id`, `action`, and `echo_data` in its payload, and the target service can return that `echo_data` in its own response payload. The router should treat payload as opaque.

This keeps the router close to a message bus:

```text
source + target + opaque payload
```

## Service Address

Addressing should be structured by conversation scope, not path-string matching:

```rust
pub struct ServiceAddress {
    pub conversation_id: ConversationId,
    pub endpoint: ServiceEndpoint,
}

pub enum ServiceEndpoint {
    ConversationControl,
    Session {
        session_id: String,
    },
    Channel {
        channel_id: String,
    },
    Workspace,
    Service {
        name: ServiceName,
    },
}
```

`ServiceName` is the dynamic part and should be a validated string:

```rust
pub struct ServiceName(String);
```

Validation rule:

- Only ASCII letters, `_`, and `-`.
- Empty names are invalid.
- No path separators, dots, spaces, digits, or other punctuation.

Examples:

```text
web-main-000001 : conversation_control
web-main-000001 : session(web-main-000001.foreground)
web-main-000001 : session(web-main-000001.background.0001)
web-main-000001 : channel(web)
web-main-000001 : channel(telegram-main)
web-main-000001 : workspace
web-main-000001 : service(memory)
web-main-000001 : service(skill)
web-main-000001 : service(cron)
web-main-000001 : service(tool_binary)
```

The global manager can route first by `conversation_id` to a conversation owner. Inside one conversation owner, the local registry routes by `ServiceEndpoint`.

`ServiceAddress` should always include `conversation_id`, even inside the conversation owner. This is not considered meaningful overhead because addresses are generated by the program, not by the LLM, and the explicit scope makes logs, forwarding, and future cross-conversation routing clearer.

## Workspace Endpoint

`ServiceEndpoint::Workspace` is an addressable workspace capability endpoint. It is not a workspace path.

The endpoint represents a workspace-facing service that can receive packets and emit packets. This is useful because workspace operations are not always simple synchronous helper calls. Some workspace operations may need to:

- Read or mutate local conversation workspace files.
- Operate on a fixed SSH remote workspace.
- Materialize local overlay files.
- Upload, download, archive, or preview large artifacts.
- Report progress to a channel endpoint.
- Return structured results to a session endpoint.

The address should not expose local/remote workspace implementation details. In fixed SSH mode, the external workspace view should behave like a single overlay filesystem-like endpoint. The workspace service decides internally whether each operation touches local overlay state, remote SSH state, or both.

The exact payload protocol for workspace operations should live in a workspace service protocol document. The router should not inspect workspace action names.

## Conversation Control Endpoint

`ServiceEndpoint::ConversationControl` means messages addressed to the conversation's own durable/control surface. It is not the router itself.

Examples of conversation-control actions:

- Change nickname.
- Switch foreground model.
- Set remote workspace.
- Set sandbox mode.
- Set reasoning effort.
- Query conversation status.
- Start/stop a managed session.
- Register/unregister a service endpoint.
- Persist state and publish conversation updates.

This keeps the distinction clear:

```text
ConversationRouter
  routes packets by target endpoint

ConversationControlService
  handles conversation-owned state and lifecycle actions
```

Both live in the same owner thread.

## Single-Threaded Owner

A possible shape:

```rust
struct ConversationOwner {
    router: ConversationRouter,
    control: ConversationControlService,
    services: ServiceRegistry,
    state: ConversationState,
}
```

The main loop receives packets and heartbeats, then routes them:

```rust
loop {
    select! {
        recv(packet_rx) -> packet => owner.handle_packet(packet?),
        recv(heartbeat_rx) -> _ => owner.handle_heartbeat(),
    }
}
```

`ConversationControlService` should not run as a separate thread. It should execute inside the owner thread because it needs to mutate durable state and update the router registry.

## Effects

Services can return explicit effects. The owner applies them serially:

```rust
enum ConversationEffect {
    Emit(ServicePacket),
    PersistState,
    PublishConversationUpdated,
    RegisterService {
        endpoint: ServiceEndpoint,
        handle: ServiceHandle,
    },
    UnregisterService {
        endpoint: ServiceEndpoint,
    },
    StartSession {
        session_id: String,
        initial: SessionInitial,
    },
    StopSession {
        session_id: String,
    },
}
```

Route table mutation happens only through owner-applied effects:

```rust
match effect {
    ConversationEffect::RegisterService { endpoint, handle } => {
        self.router.register(endpoint, handle);
    }
    ConversationEffect::StartSession { session_id, initial } => {
        let handle = self.start_session_process(&session_id, initial)?;
        self.router.register(ServiceEndpoint::Session { session_id }, handle);
    }
    ConversationEffect::Emit(packet) => {
        self.handle_packet(packet)?;
    }
    _ => { /* apply other owner-side effects */ }
}
```

This keeps the router simple while still allowing conversation-control messages to add/remove sessions and services.

## Services

Conversation-owned services can move into focused modules:

```text
stellaclaw/src/conversation/
  store.rs
  address.rs
  packet.rs
  router.rs
  effects.rs
  owner.rs
  services/
    conversation_control.rs
    session.rs
    memory.rs
    skill.rs
    cron.rs
    tool_binary.rs
    channel.rs
```

Service handlers should avoid taking `&mut ConversationRuntime` directly. Prefer a narrow context plus explicit effects:

```rust
trait ConversationService {
    fn endpoint(&self) -> ServiceEndpoint;

    fn handle(
        &mut self,
        packet: ServicePacket,
        context: ServiceContext,
    ) -> Result<Vec<ConversationEffect>>;
}
```

Slow work can later move into worker threads. Worker completion should come back to the owner as a `ServicePacket`, not mutate the registry/state directly.

For example, a future `workspace(remote)` endpoint might start a large upload/download or archive build. The owner thread should not block while the transfer runs. The workspace service can spawn a worker and immediately return. The worker can later send packets such as:

```text
source=workspace(remote)
target=channel(web)
payload={ "type": "progress", "operation_id": "...", "bytes_done": 1048576 }

source=workspace(remote)
target=session(web-main-000001.foreground)
payload={ "type": "response", "request_id": "...", "result": ... }
```

The exact payload shape belongs to the workspace service protocol.

## Service Protocol Files

The router should not standardize `request_id`, response shape, `action`, or `echo_data` globally.

Each service and the tools/clients that call it should define their own payload protocol and keep that protocol documented in one place. For example:

```text
conversation/services/memory/protocol.md
conversation/services/skill/protocol.md
conversation/services/session/protocol.md
conversation/services/workspace/protocol.md
```

This allows tightly coupled callers and target services to agree on request/response shape without making the router parse or own those semantics.

## Channels As Services

Channels should be represented as addressable endpoints from the conversation point of view:

```rust
ServiceEndpoint::Channel { channel_id: String }
```

This gives the model a consistent shape: sessions, workspaces, channels, and built-in host services all exchange `ServicePacket`s.

Current implementation does not need to expose cross-channel routing yet, but the address model should allow it later. For example, a workspace-facing endpoint could eventually send a packet to a different channel endpoint without inventing a second delivery path.

Channel services can still be backed by external channel infrastructure. The important part is that inside the conversation owner, channel delivery is represented as packets to a channel endpoint.

## Migration Direction

Suggested sequence:

1. Add `address.rs`, `packet.rs`, `router.rs`, and `effects.rs` with minimal types.
2. Keep existing `ConversationCommand`, `SessionRequest`, and `SessionEvent`, but wrap them into `ServicePacket` payloads at the conversation boundary.
3. Add `ConversationOwner` as the single-threaded owner facade around current `ConversationRuntime` behavior.
4. Register `ConversationControl`, foreground session, existing managed sessions, and built-in services during conversation startup.
5. Replace `pump_session_events + recv_timeout + keepalive` with one owner event loop that receives packets and heartbeat ticks.
6. Move service-specific handlers out of `conversation.rs` one by one.
7. After the service bus is stable, gradually reduce special `SessionRequest` / `SessionEvent` handling in favor of service payload protocols.

The first implementation should preserve current behavior and avoid changing `config` or `workdir` schema.

## Open Questions

- Should router-level routing ever inspect payload action names? Current leaning: no. If the router is intentionally a dumb bus, action names remain service-private. Router-level action inspection only becomes useful if we later need central permission policy, observability, or rate limiting by action.
- Should `Channel { channel_id }` be enough, or do we also need a channel instance/subaddress for multi-subscriber WebSocket clients?
- Which service protocol documents should be created first during migration?
